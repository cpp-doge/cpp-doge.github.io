A Matter Of Style

What Is Style

Style consists of the subjective choices you make like how you name your variables, how you organize your files, how much whitespace you use and where you put that whitespace.

Automatic Tools

You can use tools to automatically format your code.

  • Clang Format
  • Clang Tidy

These tools do a lot but they don’t do everything. You still need to observe some guidelines by hand.

Guidelines

There are many flavors of style to choose from. Try the default styles available in Clang Format to get an idea of what they are like and how you might want to customize your own.

I also recommend adopting the

  • Cpp Core Guidelines

Why And How

Consistency

Consistency breeds familiarity. Familiarity breeds quick recognition. Quick recognition frees your brain to think about more important things. All of this means you can work faster and write better more robust code.

Impact

Style transcends just the aesthetic. Style can and does have an impact on your product at the end of the day.

This can manifest in process and project management as well as in customer satisfaction and public relations.

Expectations

Setting expectations helps you write better more correct programs with fewer bugs and less pain. Keeping paradigms consistent means there is less chance for something to go wrong because there are well established patterns of development that are known to work and be effective.

Adopting standardized approaches to not only whitespace but also to file, function and class structure helps ensure a smoother better experience for everyone, and takes a lot of the guess work out of writing robust bug free code.

Just Keep It Consistent

There are many subjective decisions to be made with regard to style.

  • Same line curly braces or next line curly braces?
  • Hungarian notation?
  • Spaces inside of conditional clauses?
  • Spaces between template arguments?
  • Camel case, snake case or Pascal case?
  • East const or west const?
  • East ref or west ref?
  • East pointer or west pointer?

The answer is simple:

  • It doesn’t matter what you choose
  • Just keep it the same for everything
  • Keep it consistent

Basic Concepts

Whitespace

We put spaces between words in English to help make things easier to read.

You should do the same for your code.

Whitespace characters are things like spaces, line breaks and tabs.

In C++ these are written as

  • ' '
  • '\n'
  • '\t'

These are the “invisible” yet visible characters. They contain no squiggly lines like the letter g or the letter z but they still affect how your code looks.

When you press the space bar you insert a space. When you hit enter you insert a line break, also called a newline.

Where whitespace is allowed, you may use as much of it as you wish.

If you are allowed to put one space, you may instead put 3 or 4, or put a line break instead.

Indentation

Indentation is adding some extra whitespace at the beginning of a line.

We indent code blocks to make it more immediately obvious which code belongs to which blocks.

You should indent your code appropriately. You should also make use of whitespace like newlines.

Naming Conventions

Things like variable names and function names in C++ have some limitations, like they cannot also contain spaces.

For this reason it is common to see names written in one of a few common conventional styles.

Camel Case

Camel case is capitalizing the first letter of each word instead of using spaces between words. So named because the visual appearance resembles humps on a camel.

For example we would write “hello how are you” as helloHowAreYou.

Pascal Case

Pascal case is camel case with the first letter also capitalized.

For example we would write “this is nice” as ThisIsNice.

Pascal case is often used to signify type names so for example MyModel indicates a class and myModel indicates an instance of that class.

Snake Case

Snake case, also called “lower underscore” is placing the underscore character _ between words instead of spaces.

For example we would write “hello how are you” as hello_how_are_you.

Annotation

It is common to place some letter or word first in a variable or function name to help signifiy what kind of variable it is.

For example we might write gFoo or g_foo for a global instead of just foo.

For example we might write mFoo for a member variable instead of just foo.

This can help prevent naming conflicts between names commonly used for many things like locals, globals, parameters and members.

Mixing

Try to stick with only one or two naming conventions, but some mixing can be useful.

For example m_Foo may refer to a member function while mFoo may refer to a member variable.

For example foo_bar may refer to a key in a map, while fooBar refers to a local member variable and FooBar refers to a class.

As always, no matter your choice, just remember to keep it consistent.

Examples

Bad

Really Bad

void blarg(){ while(boop){if(foo +1){bing
;baz; bar;} }}

This is bad because

  • All the code is all squished on only two lines
  • The placement of the line break illogical
  • There are very few spaces
  • The placement of the spaces is inconsistent and illogical

Still Pretty Bad

void blarg(){
while( boop){if(foo+1 ){
bing;
baz; bar;
}
}}

This is bad because

  • The logical sections are not indented, and so are not visually obvious
  • There are still very few spaces or line breaks
  • The placement of the curly braces {}, spaces and line breaks is inconsistent, illogical and difficult to quickly and visually reason about

Good

void blarg()
{
    while (boop)
    {
        if (foo + 1)
        {
            bing;
            baz;
            bar;
        }
    }
}

This is good because

  • There is more visual separation
    • It is easier to read for the same reason that we add spaces between words in English
  • It is properly indented
    • It is clear which code is logically grouped together
  • It is easy to see that the { and } all match and line up and are correctly balanced
    • Same line braces are also fine, but they are a bit more difficult to easily and quickly visually match up
  • The usage of whitespace and indentation is consistent, making the code visually predictable
    • Predictable is easier and faster to understand and reason about
