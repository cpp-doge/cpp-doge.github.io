On Software Structure & Design

The topic of how you should structure your code is a difficult topic to teach. There are not many books or courses on this topic. Largely the topic is arguably subjective or not amazingly concrete.

If you emphasize the following general principles in your software design (many of which come from the FP world), then you will be more likely to create good software the first time, without needing to radically refactor your code, or worse, throw away your code entirely.

Important Software Design Goals

  • Simplicity
  • Testability
  • Modularity
  • Decoupling
  • Composability
  • Immutability

On Testability

If there’s one primary goal you can strive for to write better software, then that goal is testability.

Simplicity is probably the most important, but testability does the most to inform how you structure your programs.

Ask yourself, how easy is it to write tests for your code.

State, and particularly mutable state is very difficult to test for.

Tightly coupled code is difficult to test, and near impossible to write unit tests for because it necessarily requires a large amount of your code present in the test at the same time.

If your code has clear, immutable inputs and outputs, then it is easy to test. Immutability is a core premise of FP.

When striving to write testable code, you will naturally arrive at nearly all of the ideal software goals listed above.

Learning Functional Programming can help you get a good grasp of how to achieve those goals, such that your code is more and more easily testable.

A Note On FP And Ideals

Functional Programming is not perfectly well defined. FP embodies a set of ideals to strive for. Not all FP languages pursue every FP ideal, some languages are more pure or more strict than others.

For example, it’s difficult to get much done in the real world without changing things and without state. Databases are a fundamental piece of technology, they aren’t going away any time soon, and maintaining and updating state is pretty fundamental to many applications. A standard SQL database is pretty much the definition of mutable state, and this is fine. Though you could also look at some more FP leaning alternatives like Datomic.

The point is you should consider these ideals, strive for them, and then use your brain to figure out when it might make sense to depart from those ideals, provided you have a valid reason to do so.
