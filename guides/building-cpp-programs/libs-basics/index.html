Introduction

There is too much to cover in this guide. This guide presents most of the bare bones basics of working with libraries on your own, by hand, from the command line.

Expect that you will need to continue learning, researching and studying as you grow your skills and career as a programmer.

Motivation

It is critically important to understand the basics of creating and using your own libraries so that you are better and more able to use other people’s libraries with less pain.

What Are Libraries

Libraries are convenient reusable collections of code. Some libraries do not require compilation but many do.

Organizing your program into multiple separate libraries has advantages such as better reusability, better modularity, less coupling, better unit testability and probably faster compile times.

If you want to use other people’s libraries, it will help greatly to understand what libraries are and how to write, build and use them yourself.

Using libraries in your program on Linux for example is usually quite easy for you and your users by using your Linux package manager such as apt or yum.

Learning how to use libraries is mandatory if you want to be successful in general C++ programming, or at least if you prefer to avoid pain.

Link Libraries

Regardless of type, your compiler needs to know how to make a library part of your program. When referring to any library that needs to become part of your program during compilation, we will refer specifically to “Link Libraries”, meaning special files that your compiler needs to know about when you build your program.

You have already seen the separate linking phase for simple programs. You can link multiple object files into a single program.

You can also link libraries into your program including your own libraries or libraries written by other people.

Static Or Shared

Libraries come in two basic flavors

  • Static libraries
  • Shared libraries

Static Libraries

Static libraries become part of your final program and so your program can be more self contained.

Static libraries are often easier and simpler to work with and involve less pain.

Static libraries also make your program larger and may not be suitable for every program.

Using the static C++ standard library is possible but generally not recommended.

Shared Libraries

Shared libraries live as separate independent files, not part of your main program.

Your program must be able to find and load these shared libraries when you run your program.

Shared libraries have some advantages like allowing users to upgrade some library used by your program without recompiling your program.

Shared libraries facilitate features such as video game mods. Many popular games allow mods such as Skyrim and Minecraft. Mods often come in the form of shared libraries.

For example the ENB Mod for Skyrim adds cool graphical effects such as richer shadows and more intense light rays. ENB is a shared library.

Unlike static libraries, shared libraries must be installed and available on a user’s computer, otherwise your program will not run.

This may involve more complexity for you when creating installers, or more complexity for your users when they install your program.

Like static libraries, shared libraries may not be suitable for every situation.

About The Examples

The further examples in this guide make some simple assumptions.

Let’s begin by supposing that you have some code for doing video game math that you want to reuse in multiple different video games you are making.

These hypothetical files are variously named

  • algebra.cpp
  • calculus.cpp
  • physics.cpp
  • graphics.cpp

and together they (hypothetically) contain common, convenient useful math functions that we use often for video game related purposes.

The goal is to turn these files into a single library file that we can use in all our video game projects.

You are expected to be creative here, you probably have different files with different names and different functions, but this example demonstrates the basic idea.

Use your imagination. You can then apply this information to your project.

You will see how to do this in the next chapters.

Building Libraries

To build a library, the steps are similar to the discrete build from earlier in this guide where you compile and link separately.

The main difference is that instead of turning your object files into a runnable program, you turn your object files in to a library that can then be used in all the video games you work on.

This is convenient for many reasons. For example if you add a new function to your library, you can simply recompile all your video games and they will all have the new function available.

You can also test your library in isolation, apart from your actual video game projects to be sure that the library works correctly. This gives you confidence that all your games will work correctly, or at least the math parts will work correctly.
